1. Executive Summary

Mayday operates three LangGraph-based security agents that form a closed-loop detection, triage, and deception pipeline:

prompt-shield
5-Layer Defense-in-Depth
intelligent-defender
SOC Automation (1:300)
honeypot-orchestration
Multi-Cloud Deception

prompt-shield — Defense-in-depth middleware that scans all input to and output from AI agents for prompt injection attacks, PII leakage, and behavioral anomalies. Operates as a sidecar/middleware layer wrapping every deployed agent.

intelligent-defender — SOC coordinator that ingests security events from CloudWatch, GuardDuty, Splunk, and SIEM feeds; correlates against threat intelligence; triages alerts with severity scoring; executes automated containment; and dispatches deception tasks.

honeypot-orchestration — Multi-cloud deception agent that analyzes architecture via AppAcuity risk diagrams and deploys honeypots across AWS, O365, and Google Workspace. Monitors interactions and generates actionable intelligence.

Closed-Loop Architecture

prompt-shield detects threats at the agent boundary, intelligent-defender triages and responds, and honeypot-orchestration deploys deception infrastructure to gather intelligence on attacker behavior. Canary token fires feed directly into intelligent-defender as CRITICAL alerts, which can trigger automated honeypot deployment near the affected resource.

All three systems are built on LangGraph StateGraph with typed state, deterministic node execution, and checkpointing — chosen specifically for the auditability and reproducibility requirements of security operations.

2. Agentic Design Patterns & Rationale

2.1 StateGraph — Deterministic Node Execution

All three agents use langgraph.graph.StateGraph as their orchestration primitive, chosen over autonomous agent loops (e.g., ReAct with unconstrained tool use) for security-critical reasons:

  • Deterministic execution order. Each node fires in a defined sequence. In intelligent-defender: log_ingestion → threat_correlator → alert_triager → responder → reporter → deception_coordinator.
  • Typed state. Every piece of data flowing through the graph is defined as a TypedDict with explicit fields and types, making it auditable and testable.
  • Checkpointing. LangGraph's built-in checkpointing records every state transition, enabling forensic reconstruction of any security decision.

2.2 Parallel Fan-Out / Fan-In

  • prompt-shield IngressGate runs three scanners in parallel (MLClassifier, RegexScanner, PerplexityChecker), then merges scores into a single ScanResult. A failing ML classifier does not block the regex scanner from catching a known signature.
  • honeypot-orchestration fans out deployment across three platforms (AWS, O365, GWS) via conditional routing, then converges at the monitor node.

2.3 Conditional Routing

honeypot-orchestration uses conditional edges to skip platforms with no pending deployments. intelligent-defender uses severity-based routing: events scoring below the threshold (default 30) are dropped entirely.

2.4 Typed State Accumulation

Each system defines a rich TypedDict state that grows through nodes:

intelligent-defenderSecurityEvent[] → ThreatIndicator[] → Alert[] → ResponseAction[] → DeceptionTask[] → IncidentReport[]
honeypot-orchestrationAppAcuityDiagram → PlacementRecommendation[] → HoneypotDeployment[] → ThreatEvent[] → ActionableIntelligence[]

2.6 Closed-Loop Feedback

prompt-shield DETECTS → intelligent-defender TRIAGES → honeypot-orchestration DEPLOYS → ThreatEvent FEEDS BACK

2.7 Human-in-the-Loop Gates

  • intelligent-defender: Containment actions (block IP, isolate instance, revoke credentials) are recorded with status "pending" for human approval by default.
  • honeypot-orchestration: SecurityGuardrails requires human approval for MODIFY_INFRA, DESTROY, or CREDENTIAL_OP actions.
  • prompt-shield: ESCALATE action routes alerts to human SOC analysts via Slack and PagerDuty.

2.8 Security-as-Code Profiles

Per-agent YAML security profiles loaded by ProfileLoader specify scanner thresholds, tool whitelists, canary token configuration, default actions, escalation channels, and rate limits. These are version-controlled alongside application code and reviewed in PRs.

4. Integration Flow

4.1 Detection-to-Deception Pipeline

End-to-End Flow

1. prompt-shield detects injection (canary fire, regex match, ML classification)
2. AlertPayload dispatched to intelligent-defender webhook
3. intelligent-defender ingests, correlates CTI, scores alert, executes containment
4. deception_coordinator dispatches to honeypot-orchestration
5. Honeypots deployed near affected resources
6. ThreatEvent from honeypot interaction feeds back to intelligent-defender
7. prompt-shield updates scanning profiles based on new threat intel

4.2 Canary Token Fire Scenario

  1. prompt-shield EgressGate detects canary token in LLM output via CanaryScanner
  2. PolicyEngine evaluates: canary fire = automatic ESCALATE (confidence: 1.0, zero false positives)
  3. AlertPayload generated with severity: "CRITICAL", detection_layer: "canary"
  4. Alert dispatched to: intelligent-defender webhook, Slack #security-alerts, PagerDuty
  5. intelligent-defender scores event at 90+ (critical base + CTI correlation)
  6. Responder executes containment: blocks session/IP, triggers credential rotation
  7. deception_coordinator dispatches honeypot deployment near affected resource
  8. Honeypots gather intelligence on attacker's lateral movement

6. External Input Scanning Architecture

6.1 IngressGate as Universal Middleware

Every external input — webhooks, email ingestion, CRM triggers, customer chat — passes through the IngressGate before any data reaches an LLM. The IngressGate coordinates parallel scanner execution and returns a single ScanResult with an Action decision.

6.4 Canary Token Architecture

  1. Generation: canari-llm creates deterministic fake secrets (API keys, AWS keys, emails) that look authentic
  2. Injection: Embedded into agent system prompts — the LLM sees them as legitimate secrets
  3. Scanning: EgressGate checks every LLM output for exact token matches — any match is a zero false-positive signal
  4. Alerting: Canary fires trigger CRITICAL alerts via webhook + PagerDuty
  5. Rotation: Tokens rotate every 30 days (configurable per profile)
  6. Forensics: All activity stored in local SQLite and forwarded to DynamoDB

6.5 Policy Engine

  • Any canary fire: Automatic ESCALATE (confidence 1.0)
  • CRITICAL regex match: Automatic BLOCK
  • ML above threshold: BLOCK or ESCALATE per profile
  • Multiple MEDIUM detections: Weighted aggregation can trigger WARN or BLOCK
  • Below all thresholds: ALLOW

7. Key Differentiators

Zero False-Positive Detection
Canary tokens provide mathematically zero false-positive breach detection
Guardrails from Day 1
AI security baked in before any deployment node is written
Security-as-Code
YAML profiles, tool whitelists, and Colang rules in version control
Deterministic Auditability
StateGraph + checkpointing = full forensic reconstruction
Founding Design Principle

"Automation without security is dangerous. Security without automation is exhausting." — Every agent gets shield coverage. No exceptions. Every security decision is logged, traced, and metriced. Changes to security posture go through the same code review process as application changes.

Closed-Loop Intelligence

Unlike standalone detection or response tools, the three-agent pipeline creates a self-reinforcing intelligence cycle. Attacks detected by prompt-shield generate threat indicators that improve intelligent-defender's CTI correlation, which drives honeypot placement that captures attacker TTPs, which feeds back into prompt-shield's scanning profiles. Each iteration sharpens the defensive posture.

Design Principles
  1. "Automation without security is dangerous." Every agent gets shield coverage.
  2. Zero-trust agents. Treat every LLM as a potentially compromised endpoint.
  3. Defense in depth. Stack ML classifiers, canary tokens, tool validation, and behavioral monitoring.
  4. Least privilege always. Tool call whitelists enforced by shield, not by the agent itself.
  5. Detection over prevention. Perfect prevention is impossible. Optimize for fast detection.
  6. Observable by default. Every shield decision is logged, traced, and metriced.
  7. Security as code. Profiles, whitelists, and rails are version-controlled and PR-reviewed.
